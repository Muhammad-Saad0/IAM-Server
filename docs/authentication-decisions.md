# Authentication Decisions

This file records the implementation decisions made while building the core authentication model.

## Schema

- Use Flyway as the source of truth for schema changes.
- Keep OAuth-specific tables out of the first authentication schema.
- Core authentication schema contains:
  - `users`
  - `roles`
  - `user_roles`
  - `auth_events`
- `users.id` is a UUID generated by application/domain code.
- `roles.id` is a database-generated `bigint identity`.
- `auth_events.id` is a database-generated `bigint identity`.
- `user_roles` uses a composite key: `user_id + role_id`.
- `auth_events.user_id` is nullable because failed login attempts may not resolve to a real user.
- Do not store `ip_address` in `auth_events` for now.

## Entity Style

- Use shared domain/JPA entities by default.
- Keep domain behavior on entities when the behavior depends only on the entity's own state.
- Do not put repositories, `EntityManager`, controllers, Spring Security filters, or HTTP/request logic in domain entities.
- Use Lombok for safe boilerplate on entities:
  - `@Getter`
  - `@NoArgsConstructor(access = AccessLevel.PROTECTED)`
- Do not use Lombok `@Data` on entities.
- Avoid public setters on domain entities by default.
- Use constructors/factory methods to create valid entities.
- Use named domain methods for state changes, for example `lock(now)` instead of `setStatus(LOCKED)`.

## Identity Decisions

- User IDs stay as UUIDs because users are public-facing identity records that may appear in APIs, logs, events, and integrations.
- UUID user IDs also let registration create related rows/events before the user is flushed to the database.
- Role IDs use incremental database IDs because roles are internal lookup/reference data.
- Auth event IDs use incremental database IDs because audit rows are written after the workflow has enough context to log the outcome.

## Time Decisions

- Domain entities should receive timestamps from the application workflow instead of calling `Instant.now()` internally.
- This keeps related objects created in the same use case on one consistent timestamp.
- Application services should own the clock, for example via a `Clock` bean later.

## Current Entity Notes

- `User.register(email, passwordHash, now)` only builds a new `User` with registration defaults.
- `User.register(...)` does not save the user, check email uniqueness, hash passwords, assign roles, or send messages.
- `UserRole` is an explicit join entity instead of `@ManyToMany` because the assignment has its own data: `assignedAt`.
- `UserRoleId` has `equals` and `hashCode` because embedded composite keys must compare by value.
- `AuthEvent.subject` stores the attempted or resolved identifier, such as an email.
- `AuthEvent.userAgent` stores the browser/client string for audit review.
